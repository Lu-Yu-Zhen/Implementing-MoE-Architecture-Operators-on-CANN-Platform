# Implementing MoE Architecture Operators on CANN Platform

> A from-scratch implementation of Mixture of Experts (MoE) inference operators on Huawei Ascend 910B AI processor using the Ascend C operator programming model, covering a three-stage pipeline: gate routing, expert FFN, and weighted combination.

## Overview

Mixture of Experts (MoE) is a sparse activation architecture widely adopted in modern large language models (e.g., DeepSeek-MoE, Mixtral, GPT-4). Its core idea is to dynamically select a small number of expert networks for each token, thereby expanding model parameters without significantly increasing inference computation.

This project implements a complete MoE inference operator pipeline on the CANN (Compute Architecture for Neural Networks) platform using the Ascend C operator programming model, tailored to the Ascend 910B hardware characteristics (24 AI Cores, 256KB UB/core, Cube+Vector dual engines). It includes three custom operators:

| Operator | Function | Inputs | Outputs |
|----------|----------|--------|---------|
| `MoeRoutingTopK` | Gate routing + Top-K expert selection | `x`, `w_gate` | `expert_ids`, `weights` |
| `MoeExpertFFN` | Expert FFN forward (GEMM + SwiGLU + GEMM) | `x`, `expert_ids`, `weights`, `expert_weights` | `expert_out` |
| `MoeCombine` | Weighted combination of expert outputs | `expert_out`, `expert_ids`, `weights` | `y` |

Data flow:

```
x ──► [MoeRoutingTopK] ──► expert_ids, weights
 │                              │
 ├──► [MoeExpertFFN] ◄──────────┘
 │        │
 │        ▼
 │     expert_out
 │        │
 └──► [MoeCombine] ◄─────────── weights
          │
          ▼
          y
```

## Operator Details

### Kernel 1: MoeRoutingTopK — Gate Routing + Top-K Selection

**Function**: Compute gate scores for each token and select the most relevant experts via Top-K.

**Algorithm**:
1. `logits = x @ w_gate` (Cube matrix multiply, FP32 accumulation)
2. Row-wise Softmax (numerically stable: subtract max → exp → normalize)
3. Top-K selection (two-pass scan for largest/second-largest experts)
4. Top-K weight normalization (selected K weights sum to 1)
5. Cast FP32 → FP16 output

**UB Budget** (`tileS=32`, `H=2048`, `E=16`):

| Buffer | Size |
|--------|------|
| `xBuf` (x tile) | 32 × 2048 × 2 = 128 KB |
| `wGateBuf` (resident) | 2048 × 16 × 2 = 64 KB |
| `logitsBuf` (GEMM output) | 32 × 16 × 4 = 2 KB |
| Temp buffer | ≈ 16 KB |
| **Total** | **≈ 210 KB < 256 KB** ✓ |

**Multi-core scheduling**: Tasks are partitioned by `tileS`; 24 cores process the task queue in round-robin fashion.

### Kernel 2: MoeExpertFFN — Expert FFN Forward (Performance Core)

**Function**: Execute SwiGLU FFN forward computation for each selected expert.

**SwiGLU FFN Formula**:
```
gate = x @ W_gate^T          [tileTokens, intermediateSize]
up   = x @ W_up^T            [tileTokens, intermediateSize]
hidden = SiLU(gate) * up     [tileTokens, intermediateSize]   ← SwiGLU activation
out  = hidden @ W_down       [tileTokens, hiddenSize]
```

Where `SiLU(x) = x * sigmoid(x) = x / (1 + exp(-x))`.

**Two-Stage Pipeline Design** (takes max, UB reuse):

| Stage | Buffers Used | Total |
|-------|-------------|-------|
| Stage A (SwiGLU) | xBuf(64KB) + wTileBuf(64KB) + projBuf(16KB) + interBuf(8KB) + tempBuf(16KB) | ≈ 170 KB |
| Stage B (Down) | interBuf(8KB) + wTileBuf(64KB) + projBuf(16KB) + tempBuf(16KB) | ≈ 104 KB |
| **max(A, B)** | | **≈ 170 KB < 256 KB** ✓ |

**Key Optimizations**:
- **K-blocking**: Partition along `hiddenSize` dimension (`tileK=128`), accumulate after each MatmulImpl Mmad
- **Workspace staging**: Stage A SwiGLU intermediate results are written to GM workspace; Stage B reads them back for Down projection, avoiding UB overflow
- **Expert parallelism**: One task per expert (`totalTaskNum = numExperts`), multi-core round-robin
- **Weight layout**: `expert_weights[numExperts, 3*intermediateSize, hiddenSize]`, three-segment storage (gate / up / down)

### Kernel 3: MoeCombine — Weighted Combination

**Function**: Weighted sum of Top-K expert outputs to produce the final output.

**Formula**:
```
y[token, d] = Σ_k  weights[token, k] * expert_out[token, k, d]
```

**Optimization**: Perform `Muls` and `Add` directly in FP16, saving Cast overhead (combine stage has lower precision requirements). UB usage is only ~12KB, well below the 256KB limit.

## Build and Run

### Requirements

- CANN Toolkit (>= 7.0)
- CMake >= 3.16
- GCC >= 7.5 (Host side) or ccec (Kernel side, provided by CANN)
- Target hardware: Ascend 910B (Atlas A2 training card)

### Option 1: NPU Operator Build (requires CANN)

```bash
# Set CANN path (if not set)
export ASCEND_CANN_PACKAGE_PATH=/usr/local/Ascend/ascend-toolkit/latest

# Default build (ascend910b)
./build.sh

# Specify SoC version
./build.sh ascend910b

# Build artifacts in build_out/, deploy with:
./build_out/custom_opp_*.run
```

### Option 2: CPU Simulation (no CANN required)

```bash
# Only needs g++, validates three-stage pipeline algorithm correctness
./build_cpu_sim.sh
```

CPU simulation tests include:
- **Tiling parameter validation**: Checks if tiling parameters satisfy UB constraints
- **UB capacity validation**: Ensures total buffer size does not exceed 256KB
- **Accuracy comparison**: Tiled implementation vs Naive reference (cosine similarity > 0.999)
- **Performance timing**: Naive vs Tiled timing comparison

### Option 3: CMake Presets

```bash
cmake --preset default
cmake --build --preset default
```

## Key Technical Points

1. **Ascend C Programming Model**: Uses CANN-specific syntax such as `__aicore__`, `__gm__`, `TPipe`, `TBuf`, `GlobalTensor`
2. **MatmulImpl Cube Matrix Multiply**: Controls data layout via `MatmulType` template parameters (ND format, transpose flags)
3. **Manual TCubeTiling Construction**: Follows the FlashAttention approach, bypassing the auto-tiling mechanism for more precise control
4. **UB Double Buffering and Tiling**: `BUFFER_NUM=2` hides transfer latency; multi-level tiling with `tileS`/`tileTokens`/`tileIntermediate`/`tileK`
5. **Workspace Two-Stage Design**: Stage A results are written to GM, Stage B reads them back, breaking through UB capacity limits
6. **Multi-core Round-Robin Scheduling**: `taskId = coreId; taskId < totalTaskNum; taskId += coreNum` pattern for load balancing

## Default Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| `numExperts` | 16 | Number of experts |
| `topK` | 2 | Experts selected per token |
| `hiddenSize` | 2048 | Hidden dimension |

| `blockDim` | 24 | AI Cores used (910B) |

| `tileTokens` (FFN) | 16 | Tokens per expert per tile (FFN stage) |

| `tileK` | 128 | K-dimension tile size |
